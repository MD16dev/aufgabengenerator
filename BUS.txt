What is an Operating System?	"The operating system is the software layer that lies between applications and hardware.&nbsp;<br><img src=""paste-ce2fb86060d30b0406dffe64321d11c51f0eba1b.jpg""><br><span style=""color: rgb(0, 170, 255);"">User interface programs&nbsp;</span><span style=""color: rgb(255, 255, 255);"">offer a high level of abstraction, they are used by user applications to manipulate the underlying system.</span><br><span style=""color: rgb(170, 0, 0);"">Kernel </span><span style=""color: rgb(255, 255, 255);"">executes operations with a high level of privilege (superviser mode)</span>&nbsp;and directly interacts with hardware.<br><br>User applications and User interface programs use U mode.&nbsp;<br>Kernel uses S mode."
Job of the Operating System	The Operating System has two main jobs:<br><ol><li>From the user perspective: to provide easy abstaction to operate with the hardware.</li><li>From the hardware perspective: manage the resources for controlled and safe use.</li></ol>In other words, the operating system must provide a clean and simple interface to access the hardware resources and provide a safe and controlled allocation of resources of programs.
Main Operating System Services (5 managements etc.)	"These features have to be provided<br><ol><li>Program management</li><li>Memory management</li><li>File management</li><li>Network management</li><li>Human interface</li></ol><img src=""paste-96be8643d7ba2b6a431c4852459e2351254e8d8d.jpg""><br>"
Overview of the hardware	"<img src=""paste-21213c727a1b6df5be9757d60f24865d1ddce050.jpg""><br><ul><li>computing hardware the executes programs: CPU</li><li>memory to store volatile information with fast access: RAM and cashes</li><li>various device controllers to communicate with hardware devices using different specifications: disk, video, USB, network</li><li>a communication layer that enables all these components to communicate commands and data: Bus</li></ul>"
CPU	"Central Process Unit, its goal is to get the instruction from the memory, decode and execute it. Each CPU understands its ""language"" - instruction set, so they are not interoperable<br><img src=""paste-009766a22815d8fa1f752bbceca53023a3b6c7c6.jpg""><br><ul><li>Execution unit, it executes the instructions<ul>
  <li>Decoder reads an instruction and forwards to the proper execution unit</li>
  <li> Arithmetic/Logical Unit (ALU) performs integer and binary operations</li>
  <li> Floating-Point Unit (FPU) performs floating-point operations</li>
  <li>Memory Unit (MU) performs memory access</li>
  </ul>
  </li>
  <li> Registers store the data <ul>
     <li> State information (stack pointer, instruction pointer, return adress) </li>
    <li> Program values in general-purpose registers</li>
    </ul>
  </li>
  <li> Multiple levels of cache help avoiding performing redundant or costly memory accesses</li>
</ul>"
Memory, Devices and Bus	"All components act as they are instructed by the CPU. CPU reads the instructions so it acts ""by itself"".<br><ul><li>Memory is a passive device that stores the data. CPU and other devices can read from and write to it.</li><li>Other devices act as the CPU instructs</li></ul>Bus is a communication layer that links all the hardware components."
Operations that may break the system (2)	<ol><li>Modification of the state of the device</li><li>Access restricted area of the memory</li></ol>That is why the OS has to isloate these operations in order to make the system safe and secure.
Protection Domains	There are usually 2 domains/modes provided:&nbsp;<br><ul><li>supervisor mode</li><li>user mode</li></ul>They are usually enforced by the hardware features
Supervisor protection domain	It is also referred as S mode and it has the highest privilege level. This means all operations are allowed.&nbsp;<br>Program running in this domain can:<br><ul><li>Use all the instructions defiend by the CPU architecture</li><li>Directly interact with any hardware device</li><li>Access any memory location</li></ul>This domain is used by the kernel.<br>All the operations that can be performed in User mode can also be performed in Supervisor mode
User protection domain	It is also referred as U mode and it has the lowest privilege level. Some operations are restricted in this domain.<br>Programms running in this domain cannot:<br><ul><li>cannot use privileged instructions that modify the system state.</li><li>access some memory areas (e.g memory of the another program).</li></ul>Programs outside of the kernel use this mode.&nbsp;
Processor privilege levels	Protection domains are backed by processor privilege levels also known as CPU modes/states. Protection levels are also called protection rings.&nbsp;<br><ul><li>x86 provides 4, - 0 the most privilege</li><li>ARMv7 provides 3, - 2 the most privilege</li><li>ARMv8 provides 4, - 3 the most privilege</li></ul>Instruction Set Architecture specifies devision of these levels inot modes, since there are usually more rings than modes (2)
Switching between modes: 2 things system checks	Applications running in User mode may need to use a privileged feature, then they have to switch to S mode.&nbsp;<br>This switching is performed through system calls. They provide a controlled interface between user space programs and privileger OS services. This ensures that user applications perform privileged operations only through kernel code. For example, read(), write() ect.<br>System call calls the function and also checks:<br><ul><li>whether the program is allowed to perform this request</li><li>whether passes argument are valid (program is not reaching restricted memory area via forged pointer)</li></ul><br><br><br>
4 types of kernels	<ul><li>Monlithic kernels</li><li>Microkernels</li><li>Hybrid kernels</li><li>Unikernels</li></ul>The choice has impact on safety, perfomance and interface:<br><ul><li>switching modes is costly</li><li>supervisor mode is not safe<br></li><li>high level of inerfaces for programmers are in different locations (kernels in monolithic, servers ans libraries in microkernels)</li></ul>
Monolithic Kernels	"Monolithic Kernel provides a large amount of core features as well as device drivers<br><img src=""paste-da41e75548e7337796d5fe48ed0a41d00600ec49.jpg""><br>Characterictics:<br><ul><li><span style=""color: rgb(0, 255, 0);"">high level of interface</span> through system calls</li><li><span style=""color: rgb(0, 255, 0);"">good perfomance</span> when components communicate</li><li><span style=""color: rgb(255, 0, 0);"">limited safety</span>, as when one component crashes, the whole system crashes</li></ul>Examples: Linux, critical embedded systems (CISCO IOS) etc.<br><br>Monolhitc kernels can be modular and allow dynamic loading of code (drivers)."
Microkernels	"A microkernels contains only minmal set of features needed. All other services are moved to user mode as servers.<br><img src=""paste-6e36155bf29f18ef5819815434d7df00fb3b0c33.jpg""><br>Characteristics:&nbsp;<br><ul><li><span style=""background-color: rgb(54, 54, 54);""><span style=""color: rgb(0, 170, 0);"">small memory footprint</span> </span>- good choice for embedded systems</li><li><span style=""background-color: rgb(54, 54, 54); color: rgb(0, 170, 0);"">enhanced safety </span>- crashed served does not crash the whole system</li><li><span style=""background-color: rgb(54, 54, 54); color: rgb(0, 170, 0);"">adaptability </span>- servers can be replaced, rebooted easily</li><li><span style=""background-color: rgb(54, 54, 54); color: rgb(255, 0, 0);"">limited perfomance</span> - inter-process communication are costly and numerous&nbsp;</li></ul>Examples: Minix, L4 family"
Hybrid Kernels	"Hybrid architecture sits between monolithic and microkernels. It is a monolithic architecture where some components are moved to User space as servers. It is not as safe as microkernel architecture, as some components are still in the kernel.<br><img src=""paste-0e0bd64c77da01efd92db93c13d554754c04c946.jpg""><br>Example: Windows NT"
Unikernels	"Unikernels, or library operating systems, embed all the software in supervisor mode. The kernel and all user applications are running in the same privilege level.&nbsp;<br><img src=""paste-7ce60a832bc5e74a6126ebdeda807e77990af940.jpg""><br>Characteristics:<br><ul><li><span style=""color: rgb(0, 255, 0);"">High perfomance</span><span style=""color: rgb(255, 255, 255);"">: system calls become regular functions and no copies between user and kernel spaces</span></li><li><span style=""color: rgb(0, 255, 0);"">Security</span>: attack surface is minimized, easier to hardden&nbsp;</li><li><span style=""color: rgb(255, 0, 0);"">Usability</span>: hard to bulid, due to lack of features supported&nbsp;</li></ul>Examples: Unikraft"
Process: definition	"A process is the abstraction that represents an instance of a running program. It contains the set of information for program to be executed properly:<br><ul><li>Code (the binary representation of the program's instruction)</li><li>Memory (variables, stack, heap, etc.)</li><li>Register (instruction pointer, stack pointer, etc.)</li><li>Resources in use (open files, sockets, etc.)</li></ul>Process are independent entities that the OS can manipulate, choose which one and when to perform.<br><span style=""color: rgb(255, 0, 0);"">Program is a recipe (code), progress is execution of the programm</span><br>"
3 ways process can be created	"<ul><li><span style=""color: rgb(255, 255, 255);"">A&nbsp;user&nbsp;creates&nbsp;it&nbsp;by&nbsp;starting&nbsp;a&nbsp;new&nbsp;program (e.g. clicking the icon)</span></li><li><span style=""color: rgb(255, 255, 255);"">Another process creates it using a process creation system call (e.g. browser opens a pdf reader, when pdf is downloaded)</span></li><li><span style=""color: rgb(255, 255, 255);"">When the system starts (initial process to initialize the system)</span></li></ul>"
2 classes of processes depending on their purpose	"<ul><li><span style=""color: rgb(255, 255, 255);"">Foreground&nbsp;-&nbsp;interacts&nbsp;with&nbsp;the&nbsp;user&nbsp;(e.g.&nbsp;video&nbsp;player)</span><br></li><li><span style=""color: rgb(255, 255, 255);"">Background - run by themselves with no user interaction required (e.g notifications)</span></li></ul>Backround processes that always stay background are called doemons ir services"
Process Hierarchy	"<img src=""paste-db3f295f63deee9225c3f2965314e90e0f943ef0.jpg""><br>In UNIX-based OS (Linux, MacOS) process are linked into a process tree. Parent node is a process that created the process of the child node. The root ""init"" is the first process created when the system boots. The init adopts orphaned processes.<br>Windows does not keep a full tree. Parent process are given a handle on their children. This handle can be passed to another process."
3 process states&nbsp;	After creation, the process can be through 3 states:<br><ul><li>Ready (=runnable) - process can be executed and is waiting for the scheduler to allocate a CPU for it</li><li>Running - the process is being on the CPU</li><li>Blocked - the process cannot be executed because it is waiting for the event to complete</li></ul>
Process control block&nbsp;	"Process control block (PCB) is used to describe the process.<br><img src=""paste-cfba0f05a4d59248e0e209f3ccc67439e353d52b.jpg"">"
Process Memory Layout (6)	"Address space is all the linear adresses that process can access.<br><img src=""paste-fc02472765e14ceb7ff39ea79a7b7257e748262a.jpg""><br><ul><li>Kernel memory is shared between all the processes, accessible only in S mode</li><li>Text - here program binary is loaded in memory</li><li>Data contain all initialized static variable (int count = 0 - local) (static int idx = 0 - global)</li><li>BSS (block starting symbol) contains static uninitialized variables (static int i)</li><li>Heap contains dynamically allocated memory, grows towards <span style=""color: rgb(255, 0, 0);"">higher </span>adresses (e. g. malloc())</li><li>Stack contains local variables, environment variable, calling context (stack frame) (<span style=""color: rgb(255, 0, 0);"">Grows towards 0x00000000</span>)</li></ul><br><br>"
Process Registers (4)	<ul><li>Instruction Pointer (IP), or Program Counter (PC) contains the address of the next instruction to execute if not jumping</li><li>Stack Pointer (SP) contains an address of the top of the stack</li><li>General Purpose (GP) contains values or addresses used by the instructions</li><li>Page Table Pointer contains the address&nbsp; of the memory mapping</li></ul>
Context switch (4 steps)	When the OS is switching currently running process, it performs a context switch:<br><ol><li>Changes the state of the A from running to ready</li><li>Saves all the CPU registers into A's PCB</li><li>Copies all the register's from the B's PCB into CPU registers</li><li>Changes the state of the B to running&nbsp;</li></ol>
Process Isolation	Processes are isolated from each other. They do not share the open files, memory etc. if it is not stated in the program explcitely.&nbsp;
Why processes is a heavyweight meachanism	"<ul><li><span style=""color: rgb(0, 255, 0);"">Good for isolation</span></li><li><span style=""color: rgb(255, 0, 0);"">Creating a process is a costly operation</span></li><li><span style=""color: rgb(255, 0, 0);"">Communication has to be set up explicitely</span></li></ul><span style=""color: rgb(255, 255, 255);"">There are other ways to execute parallel programs</span><br>"
Threads: definition	Thread is the entity that executes the instructions. It has a program counter, registers and a stack.<br>A process contains one or several threads and threads in the same process share their memory.&nbsp;
Process and threads interacrion	Process group together memory, open files, signals, etc. Threads execute the code.&nbsp;
Shared Data in Threads (7)	<ul><li>Address space</li><li>Global variables</li><li>Open files</li><li>Child processes</li><li>Pending alarms</li><li>Signals and signal handlers</li><li>Accounting metadata</li></ul>
Private Data in Threads	<ul><li>Program counter</li><li>Registers</li><li>Stack</li><li>State</li></ul>
Implementation of the Threading in PCB	Threads are implented in the PCB in Runtime as thread tables
User threads	Threading can also be handled in U level. This allows prgrams to use thread model on OSes even if they do not supprt threading and to perform scheduling by the program not the OS.<br>It is usually provided via:<br><ul><li>Programming languages: Java, Go, Haskell etc.</li><li>Libraries: OpenMP, Cilk, greenlet etc.</li></ul>These runtime systems must perform a form of context sweetching and scheduling.<br>
Thread Mapping Model: definition and 3 models	OS can only manipulate it own threads. User-level threads have to be attached to the OS threads. This process is called Thread Maping Model.<br>There are three main model to perform it:<br><ul><li>1:1</li><li>N:1</li><li>N:M</li></ul>
1:1 Thread Mapping Model	"Each user-level thread is attached to one kernel thread. Requires OS support<br>Pros and Cons:<br><ul><li><span style=""color: rgb(0, 255, 0);"">Parallelism</span></li><li><span style=""color: rgb(0, 255, 0);"">Easy of use</span></li><li><span style=""color: rgb(255, 0, 0);"">No control over scheduling</span></li></ul><span style=""color: rgb(255, 255, 255);"">In C and Java</span><br>"
N:1 Mapping Thread Model	"All user-level thread are attached to one kernel thread. OS doesn't need to support threads. The runnig system must manage scheduling, switches etc.<br>Pros and Cons:<br><ul><li><span style=""color: rgb(0, 255, 0);"">Full scheduling control</span></li><li><span style=""color: rgb(0, 255, 0);"">Reduced context switch overhead</span></li><li><span style=""color: rgb(255, 0, 0);"">No parallelism</span></li></ul><span style=""color: rgb(255, 255, 255);"">In Pyhton, Lua, PHP</span><br>"
N:M Thread Mapping Model	"N of user threads are attached to M kernel threads. Combines benefits of 1:1 and N:1 but increaes the complexity. Often one thread per core and the schedule the user threads on them.&nbsp;<br>Pros and Cons:<br><ul><li><span style=""color: rgb(0, 255, 0);"">Parallelism</span></li><li><span style=""color: rgb(0, 255, 0);"">Control over scheduling</span></li><li><span style=""color: rgb(255, 0, 0);"">May be complex to manage for the runtime</span></li><li><span style=""color: rgb(255, 0, 0);"">Perfomance issues when user and kernel schedulers make conflict decisions</span></li></ul>In Go, Haskell, Erlang"
User threads and blocking operations: 2 ways to omit	All the user threads, sharing one kernel thread, share the state of the kernel state. So if one user thread performs a blocking operation, all the shared user threads will be blocked.&nbsp;<br>2 Ways to omit it:<br><ul><li>Replace blocking operations with asynchronous operations and wait-free algorithms. Increases code complexity</li><li>Rely on multithreading, similar to coroutines in programming. This type of user thread is called <b>fibers</b>.&nbsp;</li></ul>
The scheduler: definition, key factors for desicion-making	The scheduler is OS component that allocates CPU resorces ot the threads. It decides which thread runs on which CPU, when and for how long.&nbsp;<br>It makes the decision depending on:<br><ul><li>The system itself (amount of memory, number of CPUs, etc.)</li><li>The workload runnig on it ( interactive appllications, long running programs, etc.)</li><li>The goals to achieve (responsiveness, energy, performance, etc)</li></ul>
2 classes of applications in terms of their behavior	"<ul><li>CPU-bound (or compute-long): there are few IO-waits, long computations on the CPU</li><li>IO-bound: frequent IO waits and short computations</li></ul><img src=""paste-64f57d10a139aedaf7ff5db8406c2cba69afc968.jpg""><br>"
(Non-)preemtive schedulers	<ul><li>A preemtive scheduler may stop a running thread after some period of time (only possible with hardware clock)</li><li>A non-preemtive scheduler runs a thread until it blocks, terminates or voluntarily yields the CPU</li></ul>
3 categories of Scheduling Algorithms	<ul><li>Batch: CPU-bound (i.e. apps for business)&nbsp; → no need for preemption </li><li>Interactive: IO-bound (i.e. text editors)&nbsp; → preemption is necessary for the resonsiveness</li><li>Real time: for constained environments where deadlines should be met&nbsp; (i.e. embedded systems)→ may or may not have preemption, as environmens are usually fully controlled.&nbsp;</li></ul>
Scheduling priorities: all categories	<ul><li>Fairness: Processes should get a similar amount of CPU resource. Fairness is not equality (i.e. can be weighted depeneding on the priority).</li><li>Resource utilization: Hardware resources should be kept as busy as possible&nbsp;</li></ul>
Priorities for the Batch scheduling	<ul><li>Throughput: maximize the number of jobs completeed per unit of time</li><li>Turnaround time: minimize the duration between job submission and completion</li><li>CPU utilization: CPU should be kept as busy as possible</li></ul>
Priorities of interactive scheduling	<ul><li>Response time/latency: respond to requests quickly<br></li><li>User experience: system should be responsive to user</li></ul>
Priorites of real time scheduling	<ul><li>Meeting deadlines: the job shoud be done befire requested deadline for safety reasons</li><li>Predictability: behave in deterministic way, easy model and understand</li></ul>
Batch Scheduling: examples	<ul><li>First-Come, First-Served</li><li>shortest Job First</li><li>Shortest Remaining Time</li></ul>
First-Come, First-Served	"The First-Come, First-Served (FCFS) scheduler assigns threads to CPUs in their order of arrival, in a non-preemtive way.<br><img src=""paste-7d1f325a4281e869a1ee6ce7fdd217d010e74c81.jpg""><br>Election: choosing the head of the Runqueue<br>Runqueue: sorted list of threads, first is the oldest<br>Block: removes thread and triggers Election<br>Wake up: add the thread to the queue (at the end)<br><br>Pros and Cons:<br><ul><li><span style=""color: rgb(0, 255, 0);"">easy to implement</span></li><li><span style=""color: rgb(255, 0, 0);"">with many threads, long latencies for IO-bound threads and high turnaround time for CPU-bound threads</span></li></ul>"
Shortest Job First	"Assumes that the job's run time is known in advance. Works as FCFS, but threads are sorted by the run time.&nbsp;<br>Pros and Cons:<br><ul><li><span style=""color: rgb(0, 255, 0);"">optimal when all jobs are known in advance</span></li><li><span style=""color: rgb(255, 0, 0);"">does not adapt well to differ job arrival</span></li></ul><br>"
Shortest Remaining Time	"The Shortest Remaining Time is modified preemtive version of SJF, threads are sorted by remainnig time.<br><img src=""paste-6f41108645e1be2b4aa593a12f2edb528536c626.jpg"">"
Interactive Scheduling: examples	<ul><li>Round-Robin</li><li>Priority-based</li><li>Fair</li><li>Lottery</li></ul>
Round-Robin	"Preemtive scheduler<br>Each thread is assigned a time interval (quantum), it is allowed to run. When a thread has used up all of its quantum, it goes to the end of the list and the next threadstarts running<br><br><span style=""color: rgb(255, 0, 0);"">Is not applicible for long runqueues</span>"
Priority-based Scheduling	Every thread is assigned to some priority. The scheduler chooses the thread with the highest priority, when there is no priorities left, the scheduler moves to the priority that is smaller.&nbsp;<br>It can be preemtive to improve fairness or change priorities to not hog or starve the CPU.&nbsp;
Fair scheduler	Each thread is given the same amount of CPU time desipte priority, runtime, etc.&nbsp;
Lottery scheduler	Lottery scheduler randomly chooses the next thread. The randomness can be skewed by giving more tickets to higher priority threads.
Real Time scheduler: classes	There are two classes by the deadlines<br><ul><li>Hard real time: the deadline for the thread must be met or the system breaks (car brakes)</li><li>Soft real time: deadlines can be missed sometimes without breaking the system (video player missing some frames)</li></ul>&nbsp;There are two classes by the periodicity:<br><ul><li>periodic: the task periodically runs for some amount of time</li><li>aperiodic: the task occurs in an unprdeictable way&nbsp;</li></ul>Examples of real time scheduler: Earliest Deadline First (EDF), Deadline Monotonic, etc.
Ways of mapping memory(3)	"<ul><li>Direct Physical Mapping</li><li>Single Memory Layout</li><li>Multiprocessing:&nbsp; <ul>
   <li> Swapping Memory</li>
  <li> Hardware Memory Keys</li>
  </ul></li></ul>"
Direct Physical Mapping	"Old method. Directly managing the memory via pograms, not OS. Programs can use any memory adress, and only one process can be in memory at a time. Example:&nbsp;<br><img src=""paste-4ff72e30cf401ccff6fe3af5732be7d820d2263d.jpg"">"
Swapping Memory: limitations	If we want to swich from process A to process B<br><ol><li>We move A's memory to storage</li><li>We move B's memory from storage to RAM</li></ol>Limitations:<br><ul><li>Not applicible for multi-core systems</li><li>Coping in/from storage is costly</li></ul>
Hardware Memory Keys	Realised via Hardware Support.<br>IBM360 provided protection keys:<br><ul><li>Memory is split into fixed sized blocks each with a 4-bit key</li><li>Each process has an assigned key</li><li>When the OS allocates a block, it assigns to it the key of the process asking for memory</li><li>When the process accesses the block with different key, the fault is triggered and a crash may be caused</li></ul>With a help of Hardware Memory Keys multiple processes can simultaneously live in memory and be isolated.<br>
Static Relocation	When the program is loaded into memory, all addresses are patched on the fly by adding the base address where the program was loaded. Static relocation is slow and reqiures metadata to differentiate between addresses and regular values.
2 properties to efficiently and safely manipulate with memory	<ul><li>Protection: user programs cannot reach each other's and kernel's memory</li><li>Relocation: programs can be located anywhere in memory</li></ul>
Address Space Abstraction	An address space is an abstract view of the memory as seen by a process. They are independent from each other and may be mapped to any physical address in memory. The manipulating can be done in various ways with different hardware support.
Dynamic Relocation	Dynamic Relocation is similar to static relocation, but is fully performed in the hardware. Each process in assigned 2 registers:<br><ul><li>Base - base address where the process's address space is mapped</li><li>Limit - the size of the process' address is mapped</li></ul>When the process accesses the memory address X, the CPU automatically:<br><ol><li>addr = X + base</li><li>checks: base ≤ addr &lt; base + limit</li></ol>The process is costly
Handling Limited Memory	All process may not fit in the real memory at the same time. There are 2 main techniques to solve this problem:<br><ul><li>Swapping</li><li>Virtual Memory</li></ul>The idea is to keep recent/freuently used processes in the RAM, and push old one to the storage device.
Swapping	Swapping is the technque that consists in moving memory between RAM and storage devices. Moving from RAM is swapping out. Moving memory back is called swapping in.<br>Combining dynamic relocation with swapping and eviction allows to fit multiple processes in memory simultaneously.&nbsp;
Memory Fragmentation	It is a phenomenon that happens when on memory the free space is devided into small blocks so the big blocks cannot allocate although there is enough of total free space. In this case the OS can perform compaction to merge small free chunks into one big.&nbsp;
Cost of Reading from memory and storage	"<img src=""paste-fb75ad4f2728125b79a31344afb994d1d50a8f4f.jpg""><br>If the scheduler is 4 ms, the processes cannot be swapped in/out frequently<br>Comaction takes a long time, and requires to pause the moving processes"
Address Space Size (2 types)	<ul><li>Fixed-size address spaces: whenever a process is created or swapped in, contiguous chunk of memory of the size of the process' address space is allocated. OS can easily manage that.</li><li>Variable-size address space: if the address space can grow, the OS has to find enough place to allow it.</li></ul>
Variable-size Address Space	When the Heap and Stack touch, process is Out-of-Memory(OoM), it can be either <b>killed </b>or <b>grow&nbsp;</b>
Variable-size Address Space: grow	"<ul><li>For UNIX layout: <ul>
   <li> Move limit register to increase the size</li>
  <li> Move the stack up to thw limit</li>
  </ul></li>
<li> For Fixed-stack layout: <ul>
   <li>With fulled stack the same as UNIX </li>
  <li> With fulled heap simply push the limit</li></ul></li></ul><img src=""paste-e40e6675eed8f93c3be027ca7d32a9c8a9acbf67.jpg""><br>"
Free memory management: 2 main techniques	To know if eviction/compaction will provide enough memory, OS needs to keep track of all free memory. There are two main techniques:&nbsp;<br><ul><li>Bitmaps</li><li>Free lists</li></ul>
Bitmaps	"The memory is divided into small chunks, e.g 8 bytes. In bitmap each bit represent the state of chunk:<br><ul><li>0 - free</li><li>1 - in-use</li></ul>The OS will iterate the bitmap until it finds a large enough number of consequetive ""0"".&nbsp;<br>Desing issues:<br><ul><li>Smaller chunks →  less waste, but bitmap is large  </li>
<li> Larger bitmaps → more search time</li></ul><br>"
Free lists	The OS maintains a sorted list of allocations and fre memory chunks. Each list-element contains:<br><ul><li>State: U-used, F-free</li><li>Address: the starting address</li><li>Size of the chunk</li></ul>
Allocation Strategies (4)	<ul><li>First fit: allocator return the chunk where allocation fits, the chunk splitted into newly allocated and the free rest</li><li>Next fit: same as the first fit, but the search starts where the last one ended. Performs worser, as the beginning of the memory can stay free.</li><li>Best fit: searches for the smallest where the chunk fits. Very slow.</li><li>Worst fit: always inserts in the biggest chunk in order to avoid small rests.&nbsp;&nbsp;</li></ul>
Limitations of Dynamic Relocation	<ul><li>Addressess are not flexible, they are a single piece of memory</li><li>Fragmentation cannot be avoided, only mitigated. Compaction is costly</li><li>Swapping is extremely costly</li><li>Address spaces cannot be larger than physical memory</li></ul>
The Virtual Memory Abstraction	Core Ideas<br><ul><li>Each program has its own address space, independent on physical memory</li><li>Address spaces are split into vhunks called pages</li><li>Address spaces are addressed with virtual addresses, that can be mapped anywhere in the physical memory</li><li>An address space can be partially in the memory, not the storage</li><li>when a process uses a virtual address that is not mapped in physical memory, the OS swaps in the necessery data before letting the process continueue execution</li></ul>With hardware support, protection and relocation are provided

Interrupt: definition	Interrupt is a high-priority signal sent to the CPU to inform it that it needs to handle the urgent event on the system. Due to the high priority of the event, treatment of the interrupt interrupts the current program on the CPU to handle the interrupt.
Interrupt mechanism (5 steps)	<ol><li>Device send the interrupt request (IRQ) to the interrupt controller</li><li>Interrupt controller sends a signal to the CPU with an identifier specifying the reason of the interrupt</li><li>CPU automatically switches to the supervisor mode and jumps to its interrupt handling code. 3 classes of the interrupts	<ul><li>Hardware interrupts: sent by hardware devices through the interrupt controller</li><li>Traps or software interrupts: deliberately triggered by program code, for example to switch to the s mode and perform a system call</li><li>faluts or excpetions: not deliberately triggered by the program code, usually due to an illegal operation</li></ul><br>

If multiple IRQs are peding simultaneously, the controller decides which one has the highest priority.&nbsp;
When CPU recives interrupt signal:	<div><ol><li>Immediately halts the current program</li><li>Switches to supervisor mode</li><li>Looks up the proper interrupt service routine that can process the incoming interrupt (done through interrupt vector)</li><li>Jumps to the address of the interrupt service routine and execute it&nbsp;</li></ol></div>

Device Virtualization (4)	<ul><li>Emulation: every access to the real device is trapped by the hypervisor</li><li>Paravirtualization: driver in the guest is ware of being virualized and communicate with optimized commans with the host driver</li><li>Passthrough: hypervisor is not involved in the communication. It just maps a device to a single guest</li><li>Mediation: emulation for device control and configuration + passthrough for the data flow</li></ul>
